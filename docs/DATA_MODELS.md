# Core Data Models

This document describes the primary data models used throughout the USA Signal Bot project. All models are implemented as standard Python `dataclasses` inheriting from `BaseDomainModel` to provide timestamps, validation, and JSON serialization out of the box.

## Data Module
- **OHLCVBar**: Represents a single period of price action (Open, High, Low, Close, Volume) for a specific symbol.
- **OHLCVSeriesMetadata**: Stores descriptive information about a series of OHLCV bars (source, quality, rows).
- **DataFetchRequest / Result**: Standardized contracts for requesting and receiving market data.

## Universe Module
- **UniverseSymbol**: Represents a tradable asset (Stock or ETF) with its metadata (exchange, sector, currency).
- **UniverseDefinition**: A named collection of `UniverseSymbol` objects representing the universe of assets to scan.

## Features Module
- **FeatureRow / FeatureValue**: Stores computed features (indicators, statistics) for a symbol at a specific time.

## Strategies & Signals Module
- **Signal**: Represents an actionable trading signal generated by a strategy (side, strength, confidence).
- **SignalBatch**: A collection of signals generated in a single run.

## Paper Trading Module
- **PaperOrder**: A simulated order to be sent to the execution engine.
- **PaperFill**: Represents the execution of a `PaperOrder`.
- **PaperTrade**: Tracks a full round-trip trade (entry and exit) for performance measurement.

## Portfolio & Risk Module
- **Position**: An currently open holding in the portfolio.
- **PortfolioSnapshot**: Point-in-time view of equity, cash, and all open positions.
- **RiskLimits / RiskDecision**: Models for evaluating whether an order passes risk management rules.

## Analytics Modules
- **BacktestResult**: Contains the configuration and final metrics of a historical simulation.
- **RegimeState**: Output from market regime classification (e.g., Risk On, Trending).
- **MLModelMetadata / Prediction**: Skeletons for future machine learning integration.

*Note: Phase 4 focuses strictly on creating these definitions. No live data is fetched and no actual trading or backtesting logic is implemented yet.*
